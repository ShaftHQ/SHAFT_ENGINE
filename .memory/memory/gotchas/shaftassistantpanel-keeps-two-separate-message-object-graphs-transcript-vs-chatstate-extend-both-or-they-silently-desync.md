ShaftAssistantPanel.transcript (an AssistantTranscriptView) and ShaftAssistantPanel.chatState (a ShaftAssistantChatState) each hold their own Message object graph -- they are NOT shared references. ShaftAssistantPanel.append() mirrors every mutation into both explicitly. Discovered while implementing issue #3968 (threading an explicit Message#kind field through the panel): AssistantTranscriptView#setMessages initially re-resolved kind from role alone instead of preserving the persisted value, silently discarding it on every session reload/tab-switch, because the bulk-load path did not mirror the same field-preservation the append path had. Fixed in PR #3978 with a regression test, but the general lesson generalizes: anyone adding new per-message state to this panel (beyond kind) must update both AssistantTranscriptView's and ShaftAssistantChatState's Message handling, including any bulk-load/setMessages-style path, not just the incremental append path, or the two graphs desync silently with no compiler error and no obvious symptom until a session reload.
