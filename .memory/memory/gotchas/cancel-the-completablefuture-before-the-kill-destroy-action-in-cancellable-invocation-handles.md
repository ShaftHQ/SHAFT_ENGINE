ShaftMcpInvocation.kill() originally ran killAction (process destroyForcibly) then future.cancel(true). Destroying the process unblocks the worker thread, which can complete the future normally before cancel() runs, so kill() spuriously returned false and future.isCancelled() stayed false -- flaked AssistantLocalAgentRunnerCancellationTest only under full-suite load. Fix (PR #3770): cancel first, then killAction; CompletableFuture.cancel(true) does not interrupt the supplier thread, so the process still gets destroyed and the worker's late result is discarded. Pattern applies to any handle pairing a future with a side-effecting kill action.
