For live-execution transparency (#3546), MCP `notifications/progress` emission is feasible but not pure plumbing. The API is on the classpath (`McpSyncServerExchange` + `McpSchema.ProgressNotification` in mcp-core-2.0.0; `@McpTool`/`@McpProgressToken`/`@McpProgress` in spring-ai-mcp-annotations-2.0.0). Gotcha: shaft-mcp registers tools via plain `@Tool` + `ToolCallbacks.from(service)` (`ShaftMcpApplication.shaftTools`); plain `@Tool`/`MethodToolCallback` methods do NOT receive `McpSyncServerExchange` -- only Spring AI `@McpTool` methods get `@McpProgressToken String` + `McpSyncServerExchange` injected. So progress requires exposing the long-running tools (`CaptureService.generateReplay`, codegen `startWithOptions`, `HealerService`) as `@McpTool` and threading a progress-sink callback down into `CaptureGenerator.generate()`. IntelliJ side is graceful-degrading: `ShaftMcpStdioClient.dispatch` drops id-less frames (progress has no id) -- add a `notifications/progress` branch + token->Consumer registry; advertise capability; attach `_meta.progressToken` in `callTool`. Non-verbose local-agent progress already flows via `appendLocalAgentOutput` / `AssistantLocalAgentRunner.effectiveConsumer` StructuredStreamParser. Separately (cost a CI round, PR #3556): the Python catalog-sync gate originally keyed only on the literal `@Tool(` marker, so migrating a tool to `@McpTool` dropped the scanned count and failed 4 tests -- the scanner now treats `@McpTool(` as a tool marker too.
