When reviewing delegated UI work, a passing assertIconOnlySymmetric-style sweep across panels only proves the button's state at the moment the sweep's panel was constructed. If the same button later has .setText(...)/.setIcon(...) called on it by a runtime code path (e.g. an error-recovery handler), that runtime state is invisible to the sweep unless a dedicated test drives the panel into that state AND then re-runs the sweep against the same instance -- which delegated implementations typically don't do, since their new tests construct a fresh panel per test method. Caught this in #3626 by reading ShaftIconButtons.java's actual behavior (unconditional setText("") plus fixed 32x32 size) and cross-referencing it against showRecovery()'s later .setText() call, not by trusting the subagent's self-reported 689/684/0/0/5 green run.
