In AssistantTranscriptView verbose streaming, a 100ms min-interval throttle on full re-renders never skipped anything once each whole-buffer convertMarkdown+setText render itself cost >100ms (interval always already elapsed) -- measured 617s for a 2000-line stream, barely better than the >3m48s unthrottled baseline. The real fix was shape-aware delta append: HTMLDocument.insertString of just the new tail inside the streamed code fence (O(delta), no HTML parse), throttle kept only as slow-path fallback, forced full render at terminal paths. Lesson: throttles only help when work per tick is cheaper than the tick; for growing-buffer re-renders, change the algorithm, not the frequency. Evidence: PR #3770, AssistantTranscriptViewPerformanceTest (2s after fix).
