The `memory` CLI's `load --token-budget <number>` flag is documented as 'Advisory token target for context packaging' (memory load --help). Empirically confirmed during #3606's audit: `memory load "<real task query>" --token-budget 600` returned 38128 bytes (~9500+ tokens at ~4 bytes/token) -- roughly 15x the requested budget, and the repo's own .memory/config.json default (`memory.defaultTokenBudget: 600`) behaves identically since it's the same advisory mechanism. This is an external npm tool (`memory`/`memory.cmd`), not part of this repo's source, so it cannot be patched from here. Do not assume `memory load`/`memory search` output is bounded by the configured budget when estimating a session's token cost -- measure the actual returned byte count instead. If tighter bounding is ever needed, it would require either an upstream fix to the `memory` CLI itself or a wrapper on our side that truncates/summarizes its output, both out of scope for a guidance-budget cleanup pass.
