`scripts/mcp/install-shaft-mcp.ps1`/`.sh`'s Resolve-PythonInstallerScript/resolve_python_script had a 'reuse a co-located install_shaft_mcp.py instead of downloading' shortcut, meant so `scripts/ci/verify_shaft_mcp_installer_release.py` (running the wrapper from a real repo checkout) and local dev skip network access. But the remote 'copy command' bootstrap flow downloads the wrapper itself into $env:TEMP/$TMPDIR on every run, and that shortcut had no way to distinguish 'real repo checkout' from 'scratch temp dir with debris from an earlier run' -- any install_shaft_mcp.py ever left in that temp dir was silently reused forever, never re-downloaded, even after the real script was fixed upstream. Fixed in PR #3374 by gating the shortcut on Test-ShaftEngineRepoCheckout/is_shaft_engine_repo_checkout: only trust the co-located file when a repo-root pom.xml with <artifactId>shaft-parent</artifactId> exists two directories up. Any future SHAFT installer/bootstrap script that (a) gets downloaded fresh into a scratch/temp dir per run and (b) has a 'use a local sibling file if present' optimization needs the same repo-checkout marker check, or it's a silent staleness footgun that never self-heals and only reproduces on a machine whose temp dir happens to have debris.
