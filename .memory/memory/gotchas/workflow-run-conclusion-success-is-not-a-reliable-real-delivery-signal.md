In GitHub Actions, a workflow run concludes success as long as no job actually failed - jobs skipped via job-level if do not count against it. Once mavenCentral_cd.yml's build_release_and_deliver job was changed to skip cleanly instead of failing when the version is already on Maven Central, the whole workflow_run now reports success on every no-op push between releases, not just on real deliveries. Downstream workflows keyed on workflow_run of Maven Central Continuous Delivery plus conclusion==success (publish-intellij-plugin.yml, publish-shaft-mcp.yml) would therefore re-fire on every dependabot bump or doc merge, trying to republish the same version to JetBrains Marketplace, GHCR, and the MCP registry. Fix: repoint those two to a release published event, since mavenCentral_cd.yml's announce_release job only creates a GitHub Release on a genuine delivery. Follow-on bug found and fixed: a release created with the default GITHUB_TOKEN never emits a release event at all (GitHub anti-recursion protection), so those two workflows silently never fired even after the repoint - confirmed via zero release-event workflow runs ever recorded in the repo's history. Fixed by passing token: secrets.BOT_TOKEN (a PAT) to ncipollo/release-action in mavenCentral_cd.yml's announce_release job. publishJavaDocs.yml intentionally keeps the workflow_run conclusion==success trigger, enforced by scripts/ci/validate_maven_publication.py, because a redundant javadoc-branch push on a skipped CD run is idempotent and low risk, unlike the other two. sync-sample-projects-version.yml, which also listened for release published, was deleted as redundant: scripts/ci/prepare_release_pr.py already rewrites shaft.version in every pom.xml under the repo (including sample example projects) at release-prep time, and the other properties it synced had zero drift from root pom.xml.
