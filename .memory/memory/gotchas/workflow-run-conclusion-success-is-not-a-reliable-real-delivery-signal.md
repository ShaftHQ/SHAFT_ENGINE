In GitHub Actions, a workflow run concludes success as long as no job actually failed - jobs skipped via job-level if do not count against it. Once mavenCentral_cd.yml's build_release_and_deliver job was changed to skip cleanly instead of failing when the version is already on Maven Central, the whole workflow_run now reports success on every no-op push between releases, not just on real deliveries. Downstream workflows keyed on workflow_run of Maven Central Continuous Delivery plus conclusion==success (publish-intellij-plugin.yml, publish-shaft-mcp.yml) would therefore re-fire on every dependabot bump or doc merge, trying to republish the same version to JetBrains Marketplace, GHCR, and the MCP registry. Fix: repoint those two to a release published event, mirroring sync-sample-projects-version.yml, since mavenCentral_cd.yml's announce_release job only creates a GitHub Release on a genuine delivery. publishJavaDocs.yml intentionally keeps the workflow_run conclusion==success trigger, enforced by scripts/ci/validate_maven_publication.py, because a redundant javadoc-branch push on a skipped CD run is idempotent and low risk, unlike the other two.
