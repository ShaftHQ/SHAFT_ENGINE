The pre-#3633 AssistantLocalAgentRunner.run() called process.waitFor(timeout.toMillis(), MILLISECONDS) exactly once per run: correct only when the deadline is fixed in advance. Once the deadline needed to grow live while waiting (extended by however long a pending approval bridge has been open), a single blocking waitFor call can no longer work, because there's no way to wake it up early to recompute a new, larger deadline if the accumulated-pending value changes mid-wait. The fix: replace the single wait with a loop of short waitFor(sliceMillis, unit) calls (200ms slices here), recomputing effectiveDeadlineMillis = baseMillis + min(accumulatedPendingMillisSupplier.getAsLong(), maxExtensionMillis) once per slice and returning true the moment any slice's waitFor succeeds. Why: this is the general pattern for 'wait for X, but the deadline itself is a function of external live state' -- a fixed-duration blocking wait primitive is architecturally incompatible with that requirement no matter how the deadline value is computed. How to apply: recognize this shape (deadline depends on state that can change while waiting) before reaching for a plain blocking wait call with a precomputed duration; short-slice polling is the standard escape hatch, and 200ms is a reasonable default slice size (fine-grained enough not to visibly delay real timeout detection, coarse enough not to busy-loop).
