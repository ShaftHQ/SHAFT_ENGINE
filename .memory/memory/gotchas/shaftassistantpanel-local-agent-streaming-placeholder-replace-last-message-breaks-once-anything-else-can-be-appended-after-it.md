AssistantLocalAgentRunner's streaming flow (appendStreamingLocalAgentBubble -> appendLocalAgentOutput -> finishLocalAgentResponse/stopLocalAgentStreaming) used to assume the streaming placeholder bubble is always the transcript's LAST message, and replaced 'the last message' unconditionally (old replaceLastTranscriptAndChatState). Issue #3695 (moving Run-timeline milestones like 'Cancelling...'/'Killing...'/'Auto-approved SHAFT tool: X' into standalone chat bubbles) breaks that invariant: a milestone bubble appended between the placeholder and the eventual replace becomes the new 'last message', so the blind replace clobbers the milestone instead and leaves the true placeholder ('_Running local assistant..._') stuck on screen forever. Caught by an existing regression test (assistantKilledNonVerboseRunReplacesBarePlaceholderWithKilled) that failed for exactly this reason during the #3695 implementation. Fix: track the placeholder's message index explicitly (ShaftAssistantPanel#localAgentStreamPlaceholderMessageIndex, set in appendStreamingLocalAgentBubble, consumed/reset by replaceLocalAgentStreamPlaceholder) and replace that index specifically -- fast-path via AssistantTranscriptView#replaceLast when it's still the tail, else fall back to AssistantTranscriptView#setMessages(active.messages) to resync the whole transcript so the placeholder's own bubble (not a trailing milestone) is what visibly changes. Any future code that appends new chat-state messages during an active local-agent stream must go through this same placeholder-index-aware path, not a naive 'replace last'.
