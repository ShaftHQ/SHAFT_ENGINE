ALWAYS pass `-Dallure.automaticallyOpen=false` on every local `mvn test`: the report should be GENERATED (to debug failures) but NEVER auto-opened in a browser. `-Dallure.skip=true` is NOT a real property -- silently ignored, report still generates AND opens. `-Dallure.generateReport=false` skips generation entirely (loses the debug report). `-Dallure.automaticallyOpen=false` is correct: `AllureManager.writeAllureReport()` still runs (report created under `<module>/allure-report/*_AllureReport.html`) but `openAllureReport()` early-returns. `allure.automaticallyOpen` defaults to true, which is why reports pop open otherwise. Standard scoped invocation: `mvn -q -pl <module> test -Dtest=A,B -DheadlessExecution=true -Dgpg.skip=true -Dallure.automaticallyOpen=false`; still read surefire XML for the verdict. Real keys live in `com.shaft.properties.internal.Allure`: `allure.generateReport`, `allure.automaticallyOpen`, `allure.generateArchive`. Do not confuse `allure.automaticallyOpen` with `allure.open` (default `false` already) -- the latter is the Allure-3-CLI-native `open` option written to `allurerc.yaml`, controlling whether the `allure` CLI process itself opens the report; it is a separate, already-safe property and setting it has no effect on SHAFT's own `openAllureReport()` browser-launch path. AGENTS.md's Windows/Codex Safety section now names `allure.automaticallyOpen` explicitly as a hard requirement rather than the prior vague `-D...=false`.
