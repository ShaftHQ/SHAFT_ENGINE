The SHAFT Pilot deterministic tests (shaft-pilot-core, shaft-capture, shaft-doctor, shaft-ai, shaft-heal, shaft-mcp) and the six release-contract validators execute fully only in two places: the `release-candidate` job of shaft-pilot-release.yml (triggers only on PRs touching root `pom.xml` or the workflow's infra files) and mavenCentral_cd.yml's `build_release_and_deliver` (skipped on every main push while the reactor version is already published). Feature PRs merged between releases never run them, so a broken test or validator drift merged mid-cycle first fails on the next auto-generated release PR -- which its author (github-actions bot) cannot fix. Two regressions hit release PR #3472 this way (a validator counting all `#` chars in shaft-mcp.yml as selectors; a #3445 test assuming test JVMs never launch with `-jar` while Surefire forks via a manifest-only booter jar). How to apply: when a release PR fails, suspect recently merged main PRs, not the version bump. When adding tests/validators to pilot modules, run them locally under real Surefire forking (not just the IDE) and read the surefire XML. prepare-release-pr.yml now runs the full validator batch before creating the PR (#3472).
