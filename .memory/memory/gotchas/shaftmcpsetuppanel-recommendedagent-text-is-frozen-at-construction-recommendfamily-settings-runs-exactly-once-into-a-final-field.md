shaft-intellij's ShaftMcpSetupPanel has two setText() call sites for the recommendedAgent JLabel: the constructor and updateLiveSummary(). Both call recommendedAgentText(), which reads only from a `final` field populated once at construction via recommendFamily(settings) -- it never re-evaluates from the live assistant family/runtime combo selection. Within a single panel instance, both call sites therefore always write the identical string; a test that changes the family/runtime combo and expects recommendedAgent's text (or its accessible description) to change will see no difference, not because a live-mirroring fix is broken, but because the underlying value genuinely cannot change post-construction. To prove a live-tracking fix (e.g. an accessible-description mirror) actually works for this label, construct two separate panel instances with genuinely different real recommendation-detection outcomes (e.g. a succeeding vs. a failing readiness probe) and compare across instances, plus prove the second call site (updateLiveSummary()) doesn't desync description from text even though the mirrored value itself doesn't change on a live combo interaction.
