AssistantTranscriptView's per-message context menu (issue #3630) was wired with a custom MouseListener whose mouseReleased/mousePressed called event.isPopupTrigger() and manually invoked JPopupMenu.show(). That path is reachable only by an actual mouse popup-trigger click -- there is no way for Shift+F10 / the keyboard context-menu key to reach it, since Swing's keyboard-menu handling is installed by JComponent.setComponentPopupMenu(menu), not by adding a MouseListener yourself. Same gap class (custom low-level input listener silently skips the platform's keyboard path) as [[shaft-intellij-a-click-to-toggle-swing-row-wired-only-with-mouselistener-is-not-keyboard-reachable]] (#3601), recurring in a different file/widget. Fix: build the JPopupMenu once per component and call component.setComponentPopupMenu(menu) -- right-click and the keyboard trigger then both funnel through the same JPopupMenu instance automatically, with zero custom trigger-detection code. If a menu item's enabled state depends on live component state (e.g. Copy depends on current text selection), refresh it in a PopupMenuListener.popupMenuWillBecomeVisible callback rather than computing it once when the menu is built, since build time and show time can be far apart.
