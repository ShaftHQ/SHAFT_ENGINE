ShaftPluginScreenshotRendererTest#rendersFeatureCatalogScreenshotsWhenOutputDirectoryIsProvided starts with Assumptions.assumeFalse(outputDirectory.isBlank(), ...), gated on -Dshaft.intellij.screenshotDir. No CI workflow (.github/workflows/intellij-plugin.yml etc, which run `gradle -p shaft-intellij check`) and no plain `gradlew test` ever sets that property, so this test's body -- including every UIManager.setLookAndFeel/configureLookAndFeel call -- never executes in CI. It is effectively a manual-only "generate PR screenshot evidence" tool. Practical implication: any NEW @Test method added to this class that isn't gated the same way becomes the first code in the whole shaft-intellij module to install a real platform L&F during an ordinary test run, which can expose latent cross-test coupling (see the DefaultTreeUI and UIManager.put() persistence gotchas).
