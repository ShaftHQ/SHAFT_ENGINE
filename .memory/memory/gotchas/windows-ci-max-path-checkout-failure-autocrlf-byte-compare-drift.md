Two Windows-only CI failure modes (PR #3435). (1) MAX_PATH checkout failure: an overlong `.memory/relations/` filename makes `git checkout` itself fail on windows-2025 runners (and plain `git clone` for users without `core.longpaths`). Mitigated by 'Enable Windows long paths' steps before checkout in all Windows-capable workflows; durable fix (#3438): `scripts/ci/validate_agent_setup.py` fails any `.memory/{memory,relations}` basename over 160 chars (`memory-filename-length` code). (2) autocrlf vs byte-compare drift gates: Windows runners materialize committed text with CRLF; any generator `--check` comparing bytes against LF output reports false drift on Windows CI only, while local runs pass (freshly generator-written, never checkout-round-tripped). Fix pattern: normalize CRLF->LF in the comparison AND pin `eol=lf` in `.gitattributes`. How to apply: make any generated-file drift gate EOL-insensitive from day one and add the output path to `.gitattributes`; when a Windows CI job dies in <20s, suspect checkout itself. When a Memory object/relation id needs shortening, prefer a structured patch (`memory patch review` / `memory save --file`, then `memory check`) over hand-editing the JSON+filename.
