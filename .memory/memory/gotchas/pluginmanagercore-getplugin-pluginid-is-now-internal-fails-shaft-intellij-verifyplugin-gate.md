JetBrains marked PluginManagerCore.getPlugin(PluginId) ApiStatus.Internal on recent IntelliJ Platform versions (confirmed against IU-262.8665.81 / 2026.2). shaft-intellij/build.gradle.kts sets INTERNAL_API_USAGES as a hard verifyPlugin failure level, so any call to it fails build-and-verify CI (seen in PR #3592, introduced earlier by the #3547/#3587 plugin-upgrade-detection feature in ShaftPluginUpgradeActivity.runningPluginVersion(), which read a plugin's own version by PluginId lookup -- a common pattern many plugins use for self-reporting). Fix: get the plugin's own descriptor via checking if SomeClassInThisPlugin.class.getClassLoader() is instanceof com.intellij.ide.plugins.cl.PluginAwareClassLoader, then call .getPluginDescriptor().getVersion() -- both getPluginDescriptor() and PluginDescriptor.getVersion() are plain public (non-Internal) methods, verified directly against JetBrains' intellij-community source. No stable replacement for looking up other plugins' descriptors exists yet (JetBrains platform team said they still plan to cover a public API for that case) -- this fix only covers the by-far-most-common case of a plugin reading its own version. Related to intellij-marketplace-release-gotchas-single-use-versions-verifyplugin-strictness-manual-gallery (same strict failureLevel list catching a different problem class).
