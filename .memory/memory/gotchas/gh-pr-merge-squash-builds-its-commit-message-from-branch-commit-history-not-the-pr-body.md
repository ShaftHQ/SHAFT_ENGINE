When squash-merging with `gh pr merge --squash` (no explicit --subject/--body), GitHub composes the squash commit message from the branch's individual commit messages/subjects, not from the PR description you edited at merge time. GitHub's issue auto-close keyword scan (Closes/Fixes/Resolves #N) reads that squash commit text. A stray closing keyword left in an early commit message -- even one you later corrected only in the PR body -- will still auto-close the wrong issue on merge. Fix: before squash-merging, `git log <base>..HEAD --format=%B` and grep for closing keywords, or pass an explicit `--subject`/`--body` to `gh pr merge` that overrides the auto-generated text. Caught via issue #3708 being closed at the exact merge timestamp of PR #3710 despite the PR body deliberately avoiding the keyword.
