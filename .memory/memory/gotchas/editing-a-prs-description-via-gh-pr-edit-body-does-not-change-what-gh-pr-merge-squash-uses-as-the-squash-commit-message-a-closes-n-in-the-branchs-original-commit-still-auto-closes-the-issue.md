To stop a PR from prematurely auto-closing a linked issue (e.g. because the implementing agent's commit said Closes #N but the fix is actually incomplete), editing the PR's GitHub description text (gh pr edit --repo ... --body '...') is NOT sufficient. gh pr merge --squash defaults to using the underlying branch commit's own message (title + body) as the squash-merge commit message, not the current PR description shown on GitHub -- so if that original commit message still contains a closing keyword (Closes/Fixes/Resolves #N), GitHub parses it from the merge commit at merge time and closes the issue anyway, regardless of what the PR description now says. Observed concretely: edited PR #3977's description to read 'Progresses #3965 (does NOT close it)' specifically to prevent auto-close because the fix was only partially landed (mechanism added, but no production caller wired to use it yet) -- issue #3965 still closed the moment the PR merged, because the branch's original commit message still said 'Closes #3965'. Had to reopen the issue after the fact. The correct fix is to override the merge commit message itself, not just the PR description: pass an explicit body to gh pr merge (if supported) or amend the branch's commit message (git commit --amend) to remove/replace the closing keyword before merging, then verify by reading the actual commit that will become the merge commit (git show <branch-tip> --format='%s%n%n%b'), not by reading gh pr view's body field.
