When the Agent tool is dispatched with isolation:"worktree", the new worktree is created from the commit that was origin/main HEAD at the START of the session -- NOT from the current HEAD of whatever branch the orchestrating session has been committing to since. If a subagent's task depends on a file/API added by an earlier commit made mid-session on the shared branch, a worktree-isolated agent will not see it: it may (a) correctly detect the gap via `git merge-base --is-ancestor <commit> HEAD` and refuse to guess, or (b) silently implement against stale file content if it doesn't check. Observed during #3643 Phase 4/5 work: #3641 (needed #3640's ShaftRunConfigurationResolver.java) and #3637 (needed 5 files touched by #3635/#3636/#3639/#3640/#3642) were both dispatched to worktrees rooted at the pre-session base commit and had to be stopped/redone.
