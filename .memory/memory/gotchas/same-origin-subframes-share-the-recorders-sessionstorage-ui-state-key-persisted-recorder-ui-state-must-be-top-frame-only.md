PR #3432 release-candidate failure (fixed in commit 9b772ea09b on release/10.3.20260711): shaft-capture-recorder.js runs in EVERY browsing context via BiDi addPreloadScript, and all same-origin frames shared the single 'shaft.capture.recorder.ui' sessionStorage key. A subframe whose realm initialized AFTER the top page's first persist (~400ms 'Open ...' breadcrumb) adopted the top page's lastUrl via the ...persisted() spread, announced a phantom 'Navigate to .../frame' row on its first 500ms navigation-poll tick, and its pagehide persist raced the top frame's (children unload after the parent, so the dirty subframe write usually wins). The next page then restored the phantom row alongside its real navigation row - singleNavigationAppendsExactlyOneNavigateRow expected 1 Navigate row but got 2, edge-on-CI only (run 29160966593). Local repro dump proved it: 'Open / | Navigate to /frame | Navigate to /popup'. Fix shape: persisted() returns {} and persist() no-ops when !topLevel, and the navigation-announce poller only runs topLevel; subframe event capture still flows via in-page queue + loopback sink POST + BiDi channel, so no delivery regression. Reusable repro technique: the bug is invisible locally because the iframe loads fast and seeds from EMPTY storage (its own href). Forcing determinism required delaying the fixture iframe response (fixture '?frameDelay=900' renders iframe src '/frame?delay=900') so the subframe initializes only after the top page persists, reproducing the slow-CI initialization order on demand. Verified failing pre-fix and passing post-fix on chrome+edge headless.
