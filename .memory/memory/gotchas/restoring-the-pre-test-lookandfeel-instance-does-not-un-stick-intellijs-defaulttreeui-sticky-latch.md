SUPERSEDED (2026-07-19, PR #3789 / issue #3786 investigation): this gotcha's central claim was disproven. Restoring the pre-test LookAndFeel instance via UIManager.setLookAndFeel(original) DOES un-stick DefaultTreeUI as the active TreeUI delegate for later tests in the same JVM -- Swing discards the accumulated UIManager defaults table (including the platform's "TreeUI" -> DefaultTreeUI mapping) on every setLookAndFeel call, so a properly-wired restore is sufficient; there is no JVM-wide sticky latch that survives a real L&F switch back.

What actually caused the originally-observed failure: a full shaft-intellij suite run with -Dshaft.intellij.screenshotDir set broke ShaftTestsPanelTest because a SECOND screenshot test, ShaftRunConfigurationOverridesPanelScreenshotTest (package com.shaft.intellij.testrunner), called UIManager.setLookAndFeel(...) with NO restoration at all -- it was never wrapped with LookAndFeelIsolationExtension the way ShaftPluginScreenshotRendererTest was in PR #3787. That unguarded consumer, not a platform-level latch, is what left IntelliJLaf (and the leaked TreeUI mapping) active when ShaftTestsPanelTest ran later in the same JVM. PR #3789 fixed it by registering LookAndFeelIsolationExtension on the previously-unguarded class; two fresh full-suite runs with screenshotDir set then passed 1020/1020.
