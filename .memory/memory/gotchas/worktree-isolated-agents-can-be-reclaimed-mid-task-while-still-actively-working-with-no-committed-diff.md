A worktree-isolated coder agent (Agent tool isolation:'worktree') had its entire worktree destroyed mid-task: .git link gone, unregistered from `git worktree list`, most files vanished, while it was deep in a long-running local reproduction (repeated scoped `mvn test` runs with `-DsurefireArgLine` overrides) and had made zero file edits/commits yet. The task's branch survived (branch refs live in the shared object DB), so no work was lost, but ~140K tokens of investigation had to be redone in a fresh worktree.

Why: the harness auto-cleans a worktree it judges to have 'no changes' (per the Agent tool's own documented behavior: 'the worktree is automatically cleaned up if the agent makes no changes'). A long reproduction/investigation phase that never touches tracked files (only runs commands with CLI property overrides) looks identical to an idle/abandoned worktree to that heuristic, even though the agent is still actively working. Orphaned Java/Maven fork processes left running by an interrupted repro likely also contributed to a partial/corrupted deletion on Windows (locked files blocking full cleanup).

How to apply: for any worktree-isolated agent whose task involves an extended read-only/reproduction phase before its first file edit (e.g. TDD red-phase reproduction via command flags, not file changes), treat an early stall or a task-notification claiming to be 'waiting for background task' as a signal to check the worktree is still registered (`git worktree list`) before trusting it's fine. If a worktree does die mid-task, the fix is cheap: `rm -rf` the orphaned directory, `git worktree prune`, and relaunch fresh on the same branch name (safe to reuse if no commits landed) - feed the new agent a condensed dump of the prior agent's findings so root-cause work isn't lost. Also have long-reproduction agents clean up their own spawned processes proactively rather than leaving forks running between steps.
