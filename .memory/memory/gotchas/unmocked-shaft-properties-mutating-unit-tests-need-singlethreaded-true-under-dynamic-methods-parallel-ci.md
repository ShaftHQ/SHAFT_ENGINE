Unit tests that call real (non-mocked) helper methods mutating thread-scoped SHAFT.Properties (e.g. LambdaTestHelper.setupDesktopWebExecution/setupMobileWebExecution/setLambdaTestProperties mutating platform.executionAddress to a real LambdaTest hub URL) will bleed that mutated state into sibling test methods sharing the same worker thread under -DsetParallelMode=DYNAMIC -DsetParallel=METHODS. Symptom: sibling tests that should hit a local/mock grid instead route to the real external host and fail with connection resets (ClosedChannelException) or malformed auth challenges (missing WWW-Authenticate) after burning the full retry budget -- looks exactly like grid infra instability but isn't. Fix: @Test(singleThreaded = true) on the offending unit-test class, same pattern as issue #3928/PR #3936's audit of 4 other classes -- this class (LambdaTestHelperUnitTest) was missed by that audit because the side effect is a network-routing property, not one of SHAFT.Properties.flags. When chasing a grid/browser infra-instability failure cluster, always check whether an unrelated unmocked unit test in the same parallel run mutates global/thread-scoped SHAFT.Properties before assuming real infra flakiness.
