ShaftIconButtons.apply(JButton, Icon) unconditionally calls button.setText("") and fixes preferredSize/minimumSize/maximumSize to a hard 32x32 square, regardless of any text set before or after the call. A Sonnet-delegated implementation of #3626 built recoveryButton/testRecoveryAction via apply(...) but then called .setText("Retry"/"Restart MCP server"/"View logs") on them at runtime in showRecovery()/configureRecoveryAction() -- text that cannot render legibly in the fixed icon slot. This was invisible to the full green test suite (689/684/0/0/5) because ShaftPanelSetupTest.toolWindowButtonsAreIconOnlyAndSymmetric() and ShaftSettingsConfigurableTest's equivalent unfiltered sweep only ever exercise these buttons at construction time (before any failure state sets visible text) on a freshly-constructed panel instance -- the text-asserting tests construct a *different* panel instance driven to the failure state, so the icon-only sweep and the text assertion never collide on the same object. Established fix pattern (already used ~15 times in ShaftPanelSetupTest's filter list, and via ShaftMcpSetupPanel's private applyLabeledAction(JButton,Icon) helper which sets icon+margin without touching text/size): skip ShaftIconButtons.apply() entirely for any button whose meaning depends on dynamically-set text, call button.setIcon(...) directly instead, and add the button's accessible name to the relevant icon-only sweep test's exemption filter list with a comment explaining why an icon alone can't convey the button's meaning.
