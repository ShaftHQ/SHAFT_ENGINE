Running `mvn -pl <modules> -am test` without a `-Dtest=` filter makes Maven's `-am` (also-make) execute the `test` *phase* for every upstream reactor module too, not just the target modules -- including shaft-engine's ~2300-test suite even when nothing in the change touched it. This has crashed the JVM with a native EXCEPTION_ACCESS_VIOLATION after 15+ minutes of pure waste. Compiling with `-am` is fine (dependency classes must exist), but for the `test` goal either add `-Dtest=<SpecificClass>`, or run `mvn install -DskipTests -Dgpg.skip=true -am -pl <changed-module>` once to populate `~/.m2`, then `mvn -pl <changed-modules> test` without `-am`. For Gradle roots (shaft-intellij), use `gradle test --tests "fully.Qualified.TestClass"` instead of a bare `gradle test`. Only run genuinely repo-wide tests when the change itself is repo-wide (e.g. a shared base class or build-tool config). Mechanically enforced by `.claude/hooks/guard.py`'s R1 check, which blocks any mvn test/verify/install/deploy/package/surefire/failsafe invocation that is unscoped or uses -am without -Dtest=.
