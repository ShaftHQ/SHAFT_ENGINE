When a scripts/ci/*.py script builds a command-line string (e.g. a user-configurable --merge-command/--doctor-command template) and later tokenizes it with shlex.split(rendered), any Windows-style backslash path embedded in that string gets corrupted -- shlex.split() defaults to POSIX parsing, where a bare backslash is an escape character that consumes the following character, so target\shard-blobs\shard-1 becomes targetshard-blobsshard-1. Discovered writing scripts/ci/run_sharded_and_merge.py (issue #3347/PR #3380): str(Path(...)) on Windows produces backslash-separated paths, interpolated into a command template before shlex.split()-ing it into subprocess argv. Fix: render every Path operand with .as_posix(), not str()/default formatting, before interpolating into the template -- forward-slash paths are accepted by both Windows and POSIX toolchains (mvn, java, etc.), so this is safe with no downside. Prefer this over shlex.split(rendered, posix=False), which also changes quote handling and would break quoted -Dexec.args="..." default templates.
