Two Swing/IntelliJ-platform test-isolation traps hit while adding a regression test to shaft-intellij's ShaftPluginScreenshotRendererTest: (1) UIManager.put(key, value) writes to a "user defaults" overlay that is documented to persist across UIManager.setLookAndFeel() calls -- restoring the LookAndFeel instance alone does NOT undo custom put() overrides (e.g. Button.background); you must snapshot each touched key via UIManager.get(key) beforehand and UIManager.put(key, originalValue) afterward (put with a null value removes the override entirely, falling through to the L&F's own default). (2) com.intellij.ui.treeStructure.Tree only gets IntelliJ's custom DefaultTreeUI as its TreeUI delegate once some platform L&F (IntelliJLaf/DarculaLaf) has been installed at least once in the JVM; DefaultTreeUI#repaintPath has internal assertions that do not tolerate an L&F swap happening mid-suite from another test class. Symptom seen: ShaftTestsPanelTest's JTree.setSelectionPath calls threw AssertionError inside DefaultTreeUI#repaintPath, and ToolApprovalPromptPanelTest's button-color assertions failed because a leaked dark Button.background read as "blueish" under a simple RGB heuristic -- both caused by an earlier test in the same forked JVM installing a L&F and UIManager overrides with no cleanup. Fix pattern: capture UIManager.getLookAndFeel(), JBColor.isBright(), and UIManager.get(key) for every touched key before the test body runs; restore all of them (on the EDT, via SwingUtilities.invokeAndWait) in a finally block.
