The intellij-plugin-verifier's `-ignored-problems` CLI flag (wired from the Gradle intellij-platform plugin's `pluginVerification.ignoredProblemsFile`) only filters `CompatibilityProblem` findings (unresolved class/method/field, illegal access, abstract instantiation, etc. -- the documented list in the verifier's README). Confirmed by reading `IgnoredProblemsFilter.shouldReportProblem` and `IgnoreCondition.parseCondition` in the intellij-plugin-verifier source: it matches `problem.shortDescription` against `CompatibilityProblem` objects only. Deprecated/experimental/internal/override-only/non-extendable API *usages* are a structurally different report category with NO suppression mechanism in the tool at all -- no `-suppress-deprecated-api-usages` flag exists; `-suppress-internal-api-usages` only covers internal API from JetBrains plugins; `-mute` only covers marketplace-metadata/structural checks. A well-formed ignore-file entry for a deprecated-API finding is syntactically valid but has zero effect; verifyPlugin keeps failing. If `pluginVerification.failureLevel` includes `DEPRECATED_API_USAGES` (or the other non-CompatibilityProblem levels) and a genuine deprecated-API finding shows up against a pinned future IDE build, the only real fix is to stop calling the deprecated API from source -- e.g. shaft-intellij's ShaftAssistantPanel#saveTranscript() was rewritten to use plain Swing JFileChooser instead of FileChooserFactory.createSaveFileDialog(FileSaverDescriptor, Project), since FileSaverDescriptor had exactly one constructor (the flagged vararg one) at the 2024.3 compile baseline with no non-deprecated alternative reachable from source at that platform version.
