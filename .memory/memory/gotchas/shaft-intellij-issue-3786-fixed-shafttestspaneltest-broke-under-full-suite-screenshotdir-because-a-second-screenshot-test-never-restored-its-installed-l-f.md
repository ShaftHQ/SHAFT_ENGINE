Issue #3786 (shaft-intellij DefaultTreeUI static-latch scare) turned out NOT to be an unresettable platform latch. Root cause found by decompiling the platform jar (app-client.jar, ideaIU-2024.3) and reading com.intellij.util.ui.tree.TreeUtil.repaintPath: the AssertionError ShaftTestsPanelTest hits at setSelectionPath is really `assert EventQueue.isDispatchThread()`, thrown synchronously from a PropertyChangeListener that only com.intellij.ui.tree.ui.DefaultTreeUI (not plain BasicTreeUI) installs -- so it only fires when DefaultTreeUI is the tree's active UI delegate. A one-off diagnostic print at the failure site showed UIManager.getLookAndFeel() was STILL `IntelliJLaf` (Dark) by the time ShaftTestsPanelTest ran, meaning PR #3787's LookAndFeelIsolationExtension restore never even applied to the L&F that leaked. The actual leaker was a SECOND, previously-missed screenshot test: shaft-intellij/src/test/java/com/shaft/intellij/testrunner/ShaftRunConfigurationOverridesPanelScreenshotTest.java calls UIManager.setLookAndFeel(...) exactly like ShaftPluginScreenshotRendererTest but was never wrapped with LookAndFeelIsolationExtension in PR #3787 -- an unguarded consumer, not a platform-level defect. Fix (PR closing #3786): made LookAndFeelIsolationExtension public (shaft-intellij/src/test/java/com/shaft/intellij/ui/LookAndFeelIsolationExtension.java) so classes in other packages can reuse it, and registered it via @RegisterExtension in ShaftRunConfigurationOverridesPanelScreenshotTest with the same THEME_DEFAULT_KEYS set it already applies in applyThemeDefaults. Verified: `gradlew --offline test -Dshaft.intellij.screenshotDir=<dir>` on JDK 21 (C:\Users\Mohab\.jdks\ms-21.0.11) went from 1020 tests / 4 failed (reproduced) to 1020/1020 passing across two fresh (--rerun) full-suite runs; the ordinary `gradlew --offline test` (no screenshotDir) path also stayed green. Lesson: whenever a NEW shaft-intellij test class installs a real platform L&F via UIManager.setLookAndFeel, it MUST register LookAndFeelIsolationExtension or it will silently leak L&F/UIManager state (including the TreeUI delegate class) into every other test sharing the JVM -- grep the module for `UIManager.setLookAndFeel(` before adding one, and check every hit is either LookAndFeelIsolationExtension itself or has @RegisterExtension registered.
