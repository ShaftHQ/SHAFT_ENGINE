shaft-intellij's Gradle test JVM (plain JUnit5, no platform test fixture) has no live com.intellij.openapi.application.Application. Any production code path reachable from these tests that calls ApplicationManager.getApplication().<anything> (getService, invokeLater, getMessageBus) NPEs unless guarded. Established idiom, reused across GuidedWorkflowPanel.resolveSettings(), ShaftAssistantPanel.storedCloudKey(), ShaftPluginExecutor.getInstance() (issue #3622), and ShaftCredentialService's edtOrInline() (issue #3623): check `ApplicationManager.getApplication() == null` and fall back to a lazily-built standalone instance (a private static HeadlessHolder for services) or inline/synchronous execution (for an EDT-dispatch Executor). When adding a new applicationService/projectService getInstance() or any EDT-posting helper, apply this guard from the start rather than discovering the NPE via a full-suite regression (as happened with ShaftPanelSetupTest during #3622).

Why: the module intentionally avoids IntelliJ's heavier platform test fixtures for unit-test speed, so no real Application/PasswordSafe/MessageBus is ever present in `./gradlew test`.

How to apply: before wiring any new code path to ApplicationManager.getApplication(), search for the existing guard pattern in ShaftPluginExecutor.java or ShaftCredentialService.java and mirror it; verify with a full `./gradlew.bat test` run (not just the scoped test), since the NPE surfaces transitively through unrelated panel-construction tests.
