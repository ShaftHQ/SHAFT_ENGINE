Appium `java-client` declares hard Selenium ranges `[4.42.0, 5.0)` on selenium-api/remote-driver/support. Hard ranges resolve against live Maven Central metadata and beat dependencyManagement pins (MNG-4463), so the resolved Selenium version changes the moment a new Selenium ships -- but only for builds with cold/fresh metadata. Warm `.m2` caches (dev machines, restored CI caches) keep resolving the old version, making the failure look nondeterministic: the release gate passed while the identical commit failed Maven Central delivery twice (10.3.20260712, 2026-07-13) with `NoSuchMethodError: 'String Script.onMessage(Consumer)'` -- Selenium 4.46.0 changed that return type from `long`. shaft-engine excludes selenium from its direct java-client dep, but `appium_flutterfinder_java -> java-client` re-introduced the ranges unexcluded (fixed by excluding `io.appium:java-client` from flutterfinder). Separately, shaft-mcp's `spring-boot-dependencies` BOM downgraded Selenium to 4.43 vs the reactor pin (child BOM imports beat the parent's selenium-bom import); fixed by importing selenium-bom first in shaft-mcp's own dependencyManagement. How to apply: when a NoSuchMethodError names an org.openqa.selenium class, diff the expected descriptor against `javap` of each pinned version; check for live ranges with `mvn dependency:tree -Dverbose` plus reading the dep's actual pom in `.m2` (a version dir with only a .pom + fresh metadata is a range probing candidates). New Appium-ecosystem deps (java-client consumers) require selenium exclusions.
