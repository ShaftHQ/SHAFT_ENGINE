com.intellij.execution.testframework.sm.runner.SMTRunnerEventsListener and SMTestProxy compile and are usable in shaft-intellij (build.gradle.kts platformVersion 2024.3) without adding any bundledModule/bundledPlugin. Verified empirically: a throwaway probe class implementing SMTRunnerEventsListener compiled clean against the unmodified build.gradle.kts. These classes ship inside lib/app-client.jar, part of the base com.intellij plugin's own classpath, already pulled in transitively via the existing intellijIdea(...) + bundledPlugin("com.intellij.java") declarations -- not a separate content module. Attempting bundledModule("intellij.platform.execution.impl") (the upstream JetBrains source-tree module name for this) fails fast: 'Specified bundledModule ... doesn't exist' -- checked all 56 productModuleV2 entries in product-info.json, none cover the generic SM test-framework UI (intellij.platform.smRunner.vcs is an unrelated VCS-specific extension). Register listeners for SMTRunnerEventsListener.TEST_STATUS via <projectListeners> in plugin.xml, same pattern as existing ExecutionListener registrations -- it's a project message-bus Topic<SMTRunnerEventsListener>. Java test-locator URL scheme for SM protocol: java:suite://qualifiedClassName and java:test://qualifiedClassName/methodName (confirmed via javap disassembly of JavaTestLocator.createLocationUrl).
