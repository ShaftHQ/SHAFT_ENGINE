The SHAFT Pilot Release Candidate workflow (.github/workflows/shaft-pilot-release.yml) runs one long sequential job with no continue-on-error between steps, so a failure in an EARLY step (e.g. shaft-doctor unit tests) prevents ANY later step (e.g. the headless capture-replay browser test, or the combined-modules consumer-fixture dependency-convergence check) from ever running. Fixing one failing step commonly just reveals the NEXT pre-existing, unrelated bug further down the same job -- this happened three times in a row on PR #3344 (release/10.3.20260706): a DoctorConfidenceScorer citation-scoring bug, then a headlessExecution properties bug, then a jffi dependency-convergence conflict in shaft-sikulix's Jython dependency tree. When iterating a release-candidate branch to green, expect this pattern and budget for several rounds rather than assuming one fix will finish the job.
