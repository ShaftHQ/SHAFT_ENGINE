Confirmed again in #3632 (consistent with the #3630 popup-menu headless-testing gotcha): JComponent.requestFocusInWindow() on a non-realized/non-showing component (true of every component in shaft-intellij's ShaftPanelSetupTest suite -- no JFrame/pack()/setVisible(true)/addNotify() anywhere) does not throw, unlike e.g. JPopupMenu.show(), but it also does not move real focus and cannot be reliably observed via KeyboardFocusManager.getCurrentKeyboardFocusManager().getFocusOwner() in this suite. The correct test-design fallback used for both new #3632 tests: assert the wiring instead of the outcome -- that the flow completes without exception, that the targeted button isFocusable()/isEnabled() (the real precondition for the fix to work in a realized IDE window), and lean on an existing, unmodified regression test (secondQueuedLocalAgentApprovalShowsAfterTheFirstResolves) staying green as indirect proof that a conditional branch (like the queued-vs-not-queued focus-restoration split) took the correct code path. Why: chasing a literal focus-owner assertion in this suite produces a flaky or always-false test, not a meaningful one. How to apply: for any new headless Swing test asserting keyboard-focus behavior, assert focusable/enabled preconditions plus no-exception completion, and reuse/extend an existing state-transition regression test as the proof for branch-selection correctness, rather than trying to observe actual focus transfer.
