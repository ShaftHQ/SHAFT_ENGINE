Claude Code's --permission-mode acceptEdits only auto-approves FILE EDITS (Write/Edit). In --print (non-interactive) mode every other permission-gated tool call - Bash, PowerShell, and all MCP tools - is auto-DENIED with no way to approve it, unless the command also wires --mcp-config + --permission-prompt-tool to an approval bridge. Reproduced live against claude v2.1.203 with the real capture-integration prompt: 16 permission denials (10 Bash, 3 shaft_coding_partner_plan, guide search, guardrails) in one run, ~28 turns and $1.62 mostly burned retrying denied tools, ending in a terse final answer - the 'useless Done bubble, tokens wasted, no code generated' symptom of issue #3409 item 1. With the bridge wired alongside acceptEdits (PR #3420) the identical run had 0 denials, 13 turns, $1.03, created the Page Object + test, and ran an approved Maven compile check. Rule: launch-time auto-approve flags for local agent CLIs must be checked per tool CATEGORY, never assumed global; keep the interactive approval bridge in granted runs too. Verification pattern that caught it: script the real CLI binary with the exact production flag set plus a stub MCP-over-HTTP approval server (single POST /mcp endpoint), then diff the terminal event's permission_denials array before/after.
