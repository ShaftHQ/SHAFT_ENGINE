selenium-chrome-driver and selenium-edge-driver jars each ship their own concrete AugmenterProvider (org.openqa.selenium.chrome.AddHasCdp / org.openqa.selenium.edge.AddHasCdp) registered in META-INF/services/org.openqa.selenium.remote.AugmenterProvider, loaded automatically by `new Augmenter()`'s constructor via ServiceLoader.load(AugmenterProvider.class). Their isApplicable() predicate is ChromiumDriver.IS_CHROMIUM_BROWSER (browserName chrome/edge/opera) -- it does NOT require se:cdp capability to be present, just a chromium browserName. DriverFactoryHelper.connectToRemoteServer used to ALSO add a custom anonymous AddHasCdp augmentation for chromium browsers (issue #3732 vintage code, believing HasCdp wasn't auto-applied like HasAuthentication). With two Augmentation entries both targeting the HasCdp interface, Augmenter#augment's `augmentations` Set doesn't dedupe by interfaceClass -- both survive its isApplicable filter, and ByteBuddy's DynamicType.Builder#implement throws `IllegalStateException: Already implemented interface ... HasCdp` the moment the second `.implement(HasCdp.class)` is requested, killing EVERY chromium remote/grid session (issue #3788 Defect A -- reproduced exactly with a stubbed CommandExecutor unit test, no grid needed: RemoteWebDriver with se:cdp+browserName=chrome capabilities). Fix: never add a custom AddHasCdp augmentation; the default Augmenter() already covers chromium CDP for both local and remote/grid sessions. Only AddHasAuthentication (never auto-applied) and BiDiProvider (when enableBiDi) need explicit registration.
