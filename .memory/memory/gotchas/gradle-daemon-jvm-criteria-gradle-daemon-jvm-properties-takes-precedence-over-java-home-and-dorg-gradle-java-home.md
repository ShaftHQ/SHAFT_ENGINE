Confirmed via Gradle 9.4 docs: 'If provided, Daemon JVM criteria configuration takes precedence over JAVA_HOME and org.gradle.java.home.' A gradle/gradle-daemon-jvm.properties file with toolchainVersion=21 would therefore force EVERY invocation of that project, including CI's shaft-intellij build-and-verify job in .github/workflows/intellij-plugin.yml, which sets up only JDK 17 via actions/setup-java and runs a system-installed (non-wrapper) gradle -p shaft-intellij check buildPlugin verifyPlugin, onto JDK 21, even though that job never installs a JDK 21. Live-verified locally: settings.gradle.kts parses and runs identically under the wrapper (9.3.0) and a system Gradle 9.0.0 install (matching CI's gradle/actions/setup-gradle version), so a settings.gradle.kts-level JavaVersion.current() guard is the safe alternative -- it runs on the real Daemon JVM before any IntelliJ Platform Gradle plugin resolution/task execution, and only rejects versions strictly newer than the documented ceiling (JDK 21), leaving CI's proven JDK 17 alone.
