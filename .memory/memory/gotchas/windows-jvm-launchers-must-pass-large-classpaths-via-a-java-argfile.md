Windows CreateProcess/cmd.exe rejects command lines beyond ~8K-32K chars. Any SHAFT launcher that joins every runtime-dependency jar into one `-cp <string>` and execs a new JVM inline hits this once the dependency set is realistic -- shaft-mcp alone resolves to a ~39K-char classpath (measured via `mvn dependency:build-classpath`), so it fails 100% of the time, not just 'sometimes'. Fixed instances: (1) GeneratedTestValidator.replay's /codegen replay launch (original fix). (2) CaptureCli.defaultLaunchPrefix, which relaunches the capture recorder as a daemon process -- this is the flagship 'record your first test' flow for a freshly generated demo project, so it was the most user-visible trigger for 'please shorten the command and run again'. (3) ShaftMcpApplication.captureLaunchPrefix's fallback branch, reached whenever shaft-mcp itself is NOT running from a single `-jar` (e.g. the installed ARGS_FILE/@argfile launch mode, or any dev/IDE run) and then a user starts a capture session -- the daemon relaunch rebuilt the full classpath inline. (4) McpLauncherLocator.buildFromThinJar, the shaft-cli dev-checkout fallback for launching shaft-mcp itself when working from a local build. All four now write a Java @argfile (`-cp\n"<classpath>"\n<mainClass>`) and launch via `[java, "@" + argsFile]` instead of inlining the classpath, matching the pattern already used by McpLauncherLocator's installed-version launch path and the IntelliJ run-configuration `shortenClasspath=ARGS_FILE` templates (PR #3019) shipped in the packaged example/demo projects.
