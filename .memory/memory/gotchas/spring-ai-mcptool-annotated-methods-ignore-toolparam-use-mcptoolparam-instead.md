In shaft-mcp's CaptureService.java, capture_generate_replay is annotated @McpTool (org.springframework.ai.mcp.annotation.McpTool) instead of @Tool because it needs a live McpSyncServerExchange injected for progress-notification streaming -- it registers on a separate 'toolSpecs' Spring bean via annotation-scanning (McpJsonSchemaGenerator), not the 'shaftTools' List<ToolCallback> bean that @Tool methods use. Empirically confirmed (2026-07-17) that @ToolParam(required=false) on a @McpTool method's parameter is silently ignored -- the generated JSON schema still marked sessionPath required. The correct annotation for that path is org.springframework.ai.mcp.annotation.McpToolParam(required=..., description=...), from the spring-ai-mcp-annotations artifact. Always verify with a live probe (SpringBootTest reading callback.getToolDefinition().inputSchema() for @Tool beans, or the 'toolSpecs' bean's McpSchema.Tool#inputSchema() map for @McpTool methods) rather than assuming annotation parity between the two registration paths.
