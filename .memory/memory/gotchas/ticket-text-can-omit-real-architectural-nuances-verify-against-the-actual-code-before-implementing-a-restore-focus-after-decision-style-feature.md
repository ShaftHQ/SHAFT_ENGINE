Issue #3632's ticket text asked for composer focus to be restored after an approval decision resolves, but did not mention that ShaftAssistantPanel.resolveLocalAgentApproval has a queuedLocalAgentApprovalPrompts.poll() mechanism: when another approval prompt is already queued, next.run() immediately shows it (which focuses its OWN first decision button via the new runOnEdt(approvalPanel::focusFirstDecisionButton) wiring). Composer-focus restoration (prompt.requestFocusInWindow()) therefore must only happen in the else branch (no next prompt queued) -- doing it unconditionally would steal focus back from a newly-shown queued prompt immediately after it was given focus. This was discovered only by reading resolveLocalAgentApproval's actual tail logic, not from the ticket. Why: a plausible-sounding ticket description can describe the common case correctly while silently missing an existing multi-step state machine (the approval queue) that the fix must interact with correctly. How to apply: before implementing any 'after X happens, restore/reset Y' behavior in a class with visible queue/state-machine fields (grep for similar `.poll()`/queue usage in the same class first), trace what the code does in the 'something else is queued next' branch, not just the straightforward 'nothing else pending' branch.
