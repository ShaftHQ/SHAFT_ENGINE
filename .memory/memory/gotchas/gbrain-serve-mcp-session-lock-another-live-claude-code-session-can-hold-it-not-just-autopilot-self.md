gbrain's PGLite single-writer lock (~/.gbrain/brain.pglite/.gbrain-lock/lock) can be held by a `gbrain serve` MCP subprocess belonging to a DIFFERENT, currently-running Claude Code session on the same machine (verified via Win32_Process parent-chain: bun.exe running src/cli.ts serve -> gbrain.exe serve -> claude.exe, a separate PID from this session). It auto-reconnects within seconds of being killed if that other session is active. This is distinct from the already-documented autopilot/scheduled-task lock contention (gbrain-always-on-setup-on-windows-schtasks-pair-not-autopilot-install). Before killing a lock-holding gbrain/bun process, check its parent process chain (PowerShell Get-CimInstance Win32_Process -Filter ProcessId=<pid> | Select ParentProcessId,CommandLine, walk up parents) -- it may be doing real work for another live agent session, not a stale/orphaned daemon. Ask the user before repeatedly killing it. A stale lock from a force-killed process IS auto-reaped on the next command once the holder PID is actually dead (per the v0.42.57.0 fix), so most timeouts resolve themselves without intervention once the true holder exits.
