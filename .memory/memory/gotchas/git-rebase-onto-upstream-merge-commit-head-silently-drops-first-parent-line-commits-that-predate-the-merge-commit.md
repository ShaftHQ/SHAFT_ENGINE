When branch B was built as: [own commits X, Y] -> merge branch A into B (a real, non-squash merge commit M) -> [more own commits], and branch A later gets squash-merged into main separately, the naive fix `git rebase --onto main M HEAD` computes the replay list as `git log M..HEAD`, which excludes X and Y because they are ancestors of M (M's first parent). Those commits silently vanish from the rebased branch with no error or conflict -- the rebase looks clean but the PR's diff is missing real content (in this case, issues #3624 and #3625's fixes disappeared out of PR #3647 entirely). Symptom: after rebase, `git log --oneline origin/main..HEAD` has fewer commits than expected, or Codacy/tests reference behavior that mysteriously isn't in the diff. Fix: identify the actual first-parent-only commit list with `git log --oneline --graph <before-divergence-point>..<original-branch-tip>` to see the merge structure, find where the branch's OWN chain reconnects to the shared ancestor (not the merge commit), then `git checkout -B <branch> origin/main` and `git cherry-pick <full-ordered-list-of-first-parent-commits, skipping the merge commit and the other-parent's commits which are already in main>` instead of a single `rebase --onto`. Verify commit count against `git log --oneline <merge-base-with-original>..<original-tip>` (excluding the merge commit) before trusting the result.
