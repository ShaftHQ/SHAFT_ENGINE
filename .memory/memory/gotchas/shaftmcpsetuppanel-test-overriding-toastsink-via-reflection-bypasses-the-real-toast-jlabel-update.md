shaft-intellij's ShaftMcpSetupPanel has `private Consumer<String> toastSink = this::showToast;`. Several existing tests in ShaftPanelSetupTest.java reflectively override this field (`setField(panel, "toastSink", (Consumer<String>) toast::set)`) to capture the toast MESSAGE string without needing a visible/showing panel. That override completely replaces showToast(String) -- the real method that sets `toast.setText(...)`/`toast.getAccessibleContext().setAccessibleDescription(...)`/`toast.setVisible(true)` never runs. A test that overrides toastSink and then asserts anything about the `toast` JLabel's own text/description/visibility will silently see stale/default values (JLabel() defaults to empty text, accessible description null) with no exception -- the assertion just fails with a confusing 'expected <> but was <null>'-style diff. Fix: when a test needs to verify the toast LABEL's live state (not just capture the message text), leave toastSink at its default (do not override it) so the real showToast() runs; showToast() sets text/description before its `isShowing()` early-return, so it works fine headless. Only override toastSink when the test genuinely wants to capture/assert the message string via a callback instead of reading the label.
