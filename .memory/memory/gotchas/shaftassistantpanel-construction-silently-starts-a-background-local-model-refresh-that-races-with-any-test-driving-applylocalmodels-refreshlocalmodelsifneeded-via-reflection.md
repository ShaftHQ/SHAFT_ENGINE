Constructing a ShaftAssistantPanel on a localCli route (the default) triggers refreshLocalModelsIfNeeded() during init, which fires CompletableFuture.supplyAsync(() -> AssistantLocalAgentRunner.listModels(arguments)) on ForkJoinPool.commonPool() and, on completion, calls runOnEdt(() -> applyLocalModels(family, models)). In the Gradle unit-test JVM, ApplicationManager.getApplication() is null (per the existing async-EDT-hop gotcha), so runOnEdt falls through to SwingUtilities.invokeLater -- queuing onto the real, JVM-wide AWT Event Dispatch Thread, a singleton that is NOT scoped per test and is never flushed/awaited by these JUnit tests. ShaftPanelSetupTest.assistantLocalModelTooltipReflectsFallbackWhenCliReportsNoModels (PR #3647 / issue #3643 Phase 2 build-and-verify, 2026-07-17) failed exactly once out of 694 tests on the GitHub Actions Linux runner with an AssertionFailedError at the first assertTrue(tooltip.contains("Fallback model list")) -- meaning the tooltip had already flipped to the LIVE value at assertion time, which only construction's own background refresh callback (racing the test's own synchronous, reflective applyLocalModels(panel, family, List.of()) call) could plausibly cause. Confirmed NOT caused by Phase 2's own diff: none of the three Phase 2 commits touching ShaftPanelSetupTest.java (#3624, #3625, #3626) touched the applyLocalModels/refreshLocalModelsIfNeeded code path in ShaftAssistantPanel.java. Confirmed non-reproducible locally: ran both the isolated class (179/179 tests, Gradle test --tests ShaftPanelSetupTest --rerun) and the full module suite (694/694 tests, gradle test --rerun) twice on a local Windows machine with zero failures -- the race window is real but narrow enough that it apparently needs a slower/more-loaded CI runner to hit. Why: this is a pre-existing latent test-isolation gap in the whole 'construction kicks off async CLI probing' pattern used throughout ShaftAssistantPanel, not specific to this one test method -- any test asserting on localModel's combo contents/tooltip immediately after construction is exposed to the same race, it just happens to be benign for every other such test because they don't inspect exactly this state at exactly this narrow window. How to apply: when a shaft-intellij Swing/EDT test flakes on CI but is unreproducible locally after 2+ full-suite runs, and git-blame shows the failing test file's recent commits never touched the actual code path under test, treat it as this same class of async-EDT-hop test-isolation gap rather than randomly guessing at recent-diff causation; the real fix (not yet applied -- filed as a follow-up, see the paired GitHub issue) is to give tests a way to suppress or deterministically await the construction-triggered background refresh (e.g. a package-private constructor overload or a real ProcessLauncher stub threaded through, mirroring the pattern AssistantLocalAgentRunner.listModels(arguments, processLauncher) already offers) rather than re-running CI indefinitely.
