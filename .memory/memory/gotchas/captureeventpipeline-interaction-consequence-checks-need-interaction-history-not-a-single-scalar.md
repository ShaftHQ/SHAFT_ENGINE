PR #3586's forward-only interactionConsequence check compared each buffered navigation (in pendingNavigations) against a single mutable lastInteractionAt scalar, read at FLUSH time. When multiple interactions happen in quick succession while an earlier interaction's caused navigation still sits buffered (e.g. ENTER submits a form whose results page performs a second location.replace(), then the user clicks a result link before the buffered navigations are flushed), the later unrelated click overwrites the scalar before the earlier navigations are evaluated -- both then appear to predate their own real cause and get wrongly recorded as phantom Navigate-to steps. Reproduced consistently (2/2 runs, both Chrome and Edge) via ManagedCaptureRecorderBrowserTest#searchWithSecondReplaceNavigationRecordsOnlyUserPerformedSteps on main and on 3 other unrelated open branches the same day -- a regression of previously-closed, verified-green issue #3549. Root-caused via temporary System.err trace logging in accept()/emitNavigation() (added, observed, then removed) rather than guessing from static code reading; a magnitude-based clock-skew tolerance was considered and rejected because the INITIAL navigation shows a similar-magnitude 'negative gap' for a legitimate reason (it predates all typing) and would be wrongly suppressed by the same tolerance. Fixed in #3596 by replacing the scalar with a bounded List<Instant> recentInteractionTimestamps (pruned to INTERACTION_NAVIGATION_WINDOW), searching for the most recent interaction AT OR BEFORE the navigation's own timestamp rather than whatever is currently most recent. Verified against BOTH the original #3586 regression test and the new one across two consecutive full ManagedCaptureRecorderBrowserTest runs (-DincludeCaptureBrowserE2E) before trusting the fix.
