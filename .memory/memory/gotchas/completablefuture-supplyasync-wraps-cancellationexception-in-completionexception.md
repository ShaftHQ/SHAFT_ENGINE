When a CompletableFuture.supplyAsync (or runAsync) task throws any exception -- including CancellationException -- the JDK's AsyncSupply.run() -> completeThrowable(x) -> encodeThrowable(x) wraps it as AltResult(new CompletionException(x)) unless x is already a CompletionException. CompletableFuture.isCancelled() checks `((AltResult) result).ex instanceof CancellationException` directly against the stored ex, so a future completed this way reads isCancelled() == false, and whenComplete's raw `error` BiConsumer argument is the CompletionException wrapper, not the bare CancellationException -- so `error instanceof CancellationException` checks in downstream consumers (e.g. ShaftAssistantPanel.showAgentResult) silently miss it. Only an explicit future.cancel(true)/cancel(mayInterruptIfRunning) call (via CompletableFuture.cancel(), which does `internalComplete(new AltResult(new CancellationException()))` directly, unwrapped) produces a future whose isCancelled()/whenComplete-error read as a bare CancellationException. future.get()/join() are the one exception: their reportGet() unwrapping specifically special-cases CompletionException by unwrapping to its cause before rethrowing as ExecutionException, so get()-based cancellation detection (catch ExecutionException, check getCause()) still works either way -- only isCancelled() and whenComplete's raw error param are affected. Verified empirically: CompletableFuture.supplyAsync(() -> { throw new CancellationException(); }) then .whenComplete(...) receives err = CompletionException(CancellationException), and isCancelled() = false; by contrast an incomplete future.cancel(true) leaves isCancelled() = true and whenComplete's err as a bare CancellationException. Root-caused SHAFT_ENGINE issue #3758: ShaftMcpInvocation.kill() called killAction.run() (which unblocks the background run() thread, letting it complete the future itself via its own thrown CancellationException) BEFORE future.cancel(true) -- under executor contention (suite mode), the background thread sometimes won that race, leaving the future's isCancelled() false. Fixed by reordering kill() to call future.cancel(true) first, closing the window entirely (the background thread is still blocked when cancel() runs, so cancel() always wins and any later background completion attempt is a silently-discarded no-op CAS failure).
