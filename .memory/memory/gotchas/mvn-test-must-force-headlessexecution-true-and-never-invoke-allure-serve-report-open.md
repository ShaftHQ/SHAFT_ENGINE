Every `mvn test` invocation in this repo that can reach SHAFT-driver-based browser tests MUST explicitly force headless execution and must never trigger an Allure report-open step. Two concrete failure modes this guards against: (1) headlessExecution defaults to false in shaft-engine/src/main/resources/properties/default/custom.properties, and per gotcha.shaft-auto-scaffolded-custom-properties-silently-overrides-d-system-properties, passing only `-DheadlessExecution=true` on the command line is NOT reliably sufficient when the working directory has no pre-existing custom.properties -- the auto-scaffold-and-reload cycle in PropertiesHelper/PropertyFileManager can re-derive a stale value depending on ordering, so a real headed browser can still launch and interrupt the user's own desktop session; (2) invoking `allure serve`, `allure:report` with an open flag, or any report-viewing script after tests finishes opens a browser window unprompted, which is disruptive since SHAFT_ENGINE runs on the user's own local machine, not an isolated CI box. Concretely: pass `-DheadlessExecution=true` AND verify (or pre-seed) the actual custom.properties in play if the module/workdir is new or unfamiliar; never run `allure serve`, `allure:report -Dallure.report.open=true`, or any generate_allure_report.sh/bat-style script as part of routine verification -- leave results under target/allure-results/allure-report untouched and let the user open them explicitly if they want to.
