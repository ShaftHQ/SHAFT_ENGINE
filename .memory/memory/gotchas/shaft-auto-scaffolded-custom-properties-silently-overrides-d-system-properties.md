When a working directory has no src/main/resources/properties/custom.properties, PropertiesHelper.initializeDefaultProperties() auto-copies the bundled template (shaft-engine/src/main/resources/properties/default/custom.properties, which has headlessExecution=false) into that path, then PropertyFileManager.readCustomPropertyFiles() reloads it and calls applyLoadedPropertiesToSystem(), which does loadedFromFiles.putAll(ThreadLocalPropertiesManager.getEffectiveProperties()) before pushing to System.getProperties(). getEffectiveProperties() folds in CURRENT system properties, so whatever the JVM's -D flag said at startup wins over freshly-loaded file values for keys BOTH define -- but if the -D flag itself was wrong (e.g. inherited from an outer process's own committed custom.properties, like shaft-capture's own local-dev default of headlessExecution=false), that wrong value survives untouched. Net effect: passing -DsomeProp=value to a forked JVM that runs SHAFT-driver-based code is not reliable on its own if that JVM has no pre-existing custom.properties -- the auto-scaffold-and-reload cycle can re-derive the SAME (correct or incorrect) system-property value rather than the file's freshly-read one, depending on ordering. The robust fix is to pre-write your own custom.properties at workDirectory/src/main/resources/properties/custom.properties (so the auto-scaffold's existence check finds it and skips overwriting) with the exact desired keys, in addition to passing -D flags. Traced end-to-end while diagnosing CaptureGeneratedReplayBrowserTest failing only on Linux CI (SessionNotCreatedException: Chrome instance exited, missing X server) while passing locally on Windows (which has a real desktop, masking the non-headless launch). Fixed in shaft-capture/src/main/java/com/shaft/capture/generate/GeneratedTestValidator.java (seedHeadlessCustomProperties + hardcoded -DheadlessExecution=true instead of forwarding the outer process's own value).
