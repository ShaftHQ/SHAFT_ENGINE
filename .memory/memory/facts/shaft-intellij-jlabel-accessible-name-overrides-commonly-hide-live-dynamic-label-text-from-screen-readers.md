Swing's AccessibleJLabel.getAccessibleName() already falls back live to getText() when no explicit accessible name is set. Many shaft-intellij informational JLabels (ShaftMcpSetupPanel.setupSummary, GuidedWorkflowPanel.advancedHint, ShaftAssistantPanel's fresh-project-hint label, plus pre-existing toast/status/recoveryStatus/readyChecklist/currentAgentConfiguration) explicitly override setAccessibleName(...) with a short static string even though the real, sometimes dynamically-updated, visible text carries the actual information. This is a systemic, deliberate-looking, pre-existing convention (several of these names double as findByAccessibleName(...) test selectors in ShaftPanelSetupTest/GuidedWorkflowPanelTest) rather than a bug unique to any one commit -- do not fix a couple of instances in isolation; if addressed, do it as one consolidated pass (tracked in GitHub issue #3603) that decides whether the fix belongs on the name or on a separately-set accessibleDescription so test-id stability is preserved.
