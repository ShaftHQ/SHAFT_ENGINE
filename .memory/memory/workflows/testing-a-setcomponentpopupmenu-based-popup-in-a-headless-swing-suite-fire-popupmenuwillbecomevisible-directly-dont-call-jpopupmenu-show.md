None of shaft-intellij's Swing tests (ShaftPanelSetupTest, AssistantTranscriptViewTest, etc.) realize a real on-screen window (no JFrame/pack()/setVisible(true)/addNotify() anywhere in the suite). JPopupMenu.show(invoker, x, y) calls invoker.getLocationOnScreen() internally, which throws IllegalComponentStateException when the invoker isn't actually showing on a real screen -- so it cannot be used to simulate a popup trigger in these tests. Instead, get the installed menu via component.getComponentPopupMenu(), then iterate menu.getPopupMenuListeners() and call listener.popupMenuWillBecomeVisible(new PopupMenuEvent(menu)) directly. This fires the exact same lifecycle callback Swing's BasicPopupMenuUI invokes right before painting the popup for either a mouse or keyboard trigger, without requiring screen realization -- and since a single JPopupMenu instance backs both trigger paths once wired via setComponentPopupMenu, this one simulation covers both mouse-trigger and keyboard-trigger test scenarios.
