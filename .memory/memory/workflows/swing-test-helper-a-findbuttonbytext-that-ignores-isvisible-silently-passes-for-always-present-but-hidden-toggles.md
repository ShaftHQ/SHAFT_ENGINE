shaft-intellij's pre-existing findButtonByText() test helper (AssistantTranscriptViewTest) never checks isVisible(), which was correct for the raw-evidence disclosure toggle (only ever *added* to the tree when evidence exists -- never present-but-hidden). The new #3629 collapse toggle is architecturally different: it is *always* present in the component tree (so updateCollapseState() can flip visibility as streamed content grows past the threshold without restructuring the hierarchy), just conditionally hidden via setVisible(false). Reusing findButtonByText() for it made two tests pass for the wrong reason (they found the invisible button and asserted non-null when a real user would see nothing). Fix: add a findVisibleButtonByText() variant that additionally requires button.isVisible(), and use it for any component whose presence-in-tree and user-visibility can diverge. Rule of thumb: before reusing an existing Swing lookup helper for a new component, check whether that component is ever added-but-hidden vs conditionally-added -- the two need different assertions.
