ShaftAssistantPanel.writeTranscriptToFile(Path) (#3631) wraps ApplicationManager.getApplication().executeOnPooledThread(...) around a separately-extracted method writeTranscriptSync(Path, String) throws IOException that contains just the Files.writeString(...) call and nothing else (no threading, no notifications). This lets tests invoke writeTranscriptSync directly via reflection and assert on file content deterministically and synchronously, without polling a background pooled thread or inventing an async-wait test idiom. Why: any 'do real work on a background thread, then notify on EDT' method is otherwise hard to test headlessly without flakiness. How to apply: when adding a new off-EDT (or off-main-thread in general) write/IO method to a Swing/IntelliJ-plugin class, extract the synchronous core into its own small package-private method before wrapping it in the threading/notification boilerplate -- write the seam-extraction as part of the initial implementation, not as an afterthought once tests turn out hard to write.
