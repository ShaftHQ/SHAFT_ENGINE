Before writing the #3631 (save transcript) implementation spec, the actual IntelliJ Platform SDK 2024.3 jars on disk were inspected directly (under the Gradle transform cache, e.g. .../ideaIU-2024.3-win/lib/) using `jar tf <jar>` to find which jar a class lives in and `javap -classpath <jar> <class>` to confirm exact package paths and method signatures, rather than relying on possibly-stale memory of the SDK API. This caught two things memory would likely have gotten wrong: VirtualFileWrapper lives in com.intellij.openapi.vfs (not com.intellij.openapi.ui, where a superficially similar FileChooser-adjacent class might be guessed to live), and FileSaverDialog has a save(java.nio.file.Path, String) overload that avoids needing a VirtualFile base-directory lookup. Why: SDK APIs move between packages/versions and training-data memory of them goes stale silently -- a wrong import compiles-fails immediately (cheap), but a wrong assumption about an overload's existence can waste a whole implementation path. How to apply: before specifying or implementing against any unfamiliar IntelliJ Platform SDK (or other external API surface with real jars on disk) class or method, do a quick jar tf / javap probe first -- this is the act-as-fable 'live-probe before implementing against anything external' principle applied concretely to IDE-plugin work.
