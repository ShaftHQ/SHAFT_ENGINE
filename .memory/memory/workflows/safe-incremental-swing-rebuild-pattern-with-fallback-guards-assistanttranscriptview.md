When converting a full-rebuild-per-mutation Swing render path (removeAll()+rebuild every component) to incremental updates, keep a 1:1 handle list synced with the backing data model (RenderedBubble per Message), then guard the incremental path with explicit fallback-to-full-rebuild checks for every edge case that would make in-place mutation unsafe: a pending transient widget row present, handle-list/data-model size drift, a new item landing exactly at a structural divider index, or an unexpected identity change like a role swap. Falling back to the old full-rebuild code path in those rare cases keeps correctness airtight while the realistic hot path (e.g. streamed line-by-line replaceLast calls) becomes O(1). Prove the perf win with a deterministic component-creation counter exposed via a package-private test hook, not wall-clock timing.
