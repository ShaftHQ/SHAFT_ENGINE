AssistantGeminiLiveE2ETest command shape: `gradle test --tests ...AssistantGeminiLiveE2ETest -Dshaft.intellij.liveGemini=true -Dshaft.intellij.liveMcpCommand="<java> @<shaft-mcp.args>" -Dshaft.intellij.workspaceRoot=<dir> -Dshaft.intellij.liveGeminiModel=gemini-2.5-flash` with GEMINI_API_KEY in env (never written to files). The installed shaft-mcp args file pins -Dshaft.mcp.workspaceRoot to %LOCALAPPDATA%/ShaftHQ/shaft-mcp/work, and the system property beats the SHAFT_MCP_WORKSPACE_ROOT env var -- pass that work dir as workspaceRoot (or scope the args file like ShaftMcpProjectScope does) or the MCP rejects 'Autobot working directory is outside the MCP workspace'. Every shaft.intellij.* test property must be explicitly forwarded in shaft-intellij/build.gradle.kts's test { systemProperty(...) } -- unforwarded -D flags silently fall back to defaults inside the test JVM. Google load-sheds gemini-3.5-flash structured-output (responseSchema) calls with 503 'high demand' while trivial payloads/model-list calls return 200; use gemini-2.5-flash for deterministic-ish live runs and expect occasional retries anyway. Reasoning models spend thinking tokens from maxOutputTokens -- a too-small budget truncates full generated classes into malformed JSON (INVALID_RESPONSE); AutobotService.runProviderChat raised this to 8k in PR #3371, but an already-installed MCP binary keeps the old budget until a release ships. Headless Swing screenshot rendering paints placeholder glyphs unless IconLoader.activate() runs first (see ShaftPluginScreenshotRendererTest's static block).
