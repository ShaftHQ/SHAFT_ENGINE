When a SHAFT browser test fails consistently on CI but passes locally on Windows: (1) rule out resource exhaustion first and cheaply -- add a temporary `df -h && free -h && ps aux | grep chrome` step guarded by `if: failure()` in the workflow; if resources are ample, stop chasing that theory. (2) Capture the forked/replay subprocess's OWN console output (it is often discarded with a @TempDir) by widening the failure diagnostics to read that process's redirected stdout/stderr log. (3) If the failure is a Selenium SessionNotCreatedException, add `-Dwebdriver.chrome.verboseLogging=true -Dwebdriver.chrome.logfile=<path>` to the subprocess's own command line -- Selenium's default ChromeDriverService reads these as system properties with zero SHAFT/production code changes needed, and the log shows the exact `Launching chrome: ...` command plus any `ERROR:ui/ozone/...Missing X server or $DISPLAY`-style crash reason. (4) To binary-search WHERE a config value goes wrong without more CI round-trips, reproduce locally with a fixed (non-@TempDir, non-auto-cleaned) directory: temporarily replace `@TempDir Path temp;` with `Path temp = Path.of("<fixed path>");` in the test, rerun, inspect the surviving artifacts (e.g. chromedriver.log, custom.properties) directly, then `git checkout --` the test file to revert the hack once done. (5) Add temporary `System.out.println("DEBUG_X=" + ...)` prints at each suspected mutation point in production code, rebuild just that module with `mvn -pl <module> install -DskipTests`, rerun, grep the debug lines, then revert with `git checkout --` once the root cause is found -- do not leave debug prints in committed code.
