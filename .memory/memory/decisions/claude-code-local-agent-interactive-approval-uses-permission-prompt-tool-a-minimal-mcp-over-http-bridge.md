Empirically validated against the real claude CLI (v2.1.201, scripted as a subprocess before writing any code): the raw --print --input-format stream-json --output-format stream-json stdio channel does NOT carry control_request/can_use_tool events for an externally-driven process -- those are part of an internal Remote Control/SDK session-bridge protocol not exposed over plain --print stdio. --permission-mode manual does not change this and does not even take effect in --print mode. The REAL, working mechanism is --permission-prompt-tool <mcp-tool-name>, a flag present in the CLI binary's help strings but omitted from --help output in this build. It takes an MCP tool name (mcp__<server>__<tool>) from a server connected via --mcp-config (file path or inline JSON string both work, but a temp FILE is used here to sidestep any embedded-quote argv issues on Windows .cmd shims). The CLI's own built-in safety classifier decides per-tool-call whether to auto-allow (obviously safe, e.g. Bash echo), auto-deny, or ask -- only 'ask' calls the permission-prompt-tool, via a normal tools/call JSON-RPC request with arguments {tool_name, input, tool_use_id}. The tool must reply {"content":[{"type":"text","text":"<json>"}]} where the embedded JSON is {"behavior":"allow","updatedInput":{...}} or {"behavior":"deny","message":"..."}. A MINIMAL, STATELESS HTTP-transport MCP server is sufficient: single POST /mcp endpoint, synchronous application/json responses (no SSE), no Mcp-Session-Id tracking needed (the CLI never requested one), GET rejected with 405 is fine. No --input-format stream-json or held-open stdin is needed at all -- this is a separate side channel, orthogonal to the existing stdout-reading/display pipe. Implemented as LocalAgentApprovalBridge (shaft-intellij/src/main/java/com/shaft/intellij/approval/LocalAgentApprovalBridge.java) using JDK-builtin com.sun.net.httpserver.HttpServer (no new dependency), wired into AssistantLocalAgentRunner.claudeCommand()/run() and ShaftAssistantPanel (renders the existing ToolApprovalPromptPanel widget under a local-agent: key namespace so it can never collide with SHAFT MCP tool-name approvals). The Java implementation itself was re-validated end-to-end against the real CLI (both allow and deny paths, real API calls, real file writes) via a throwaway, not-committed live-probe test before wiring it in.
