Because Codex's --json schema cannot distinguish a sandbox/approval denial from a plain execution failure (see the companion fact memory), Codex tool failures (status "failed", non-zero exit_code, or a present error object on command_execution/mcp_tool_call/tool_call items) are merged into the same permissionDenialsByTool map Claude's genuine permission_denials populate, but activitySummary() labels the footer line "Failed or denied tool calls" (not Claude's "Denied tool calls") and gives a Codex-accurate hint ("re-run with Allow source edits enabled", not "approve them when prompted", since Codex has no interactive approval prompt at all). file_change items only credit filesTouched when status=="completed"; a failed patch is counted as a denial (key "file_change") instead of a false file-edit claim. This was an explicit, documented tradeoff per the issue's own suggested resolution, chosen over inventing a new footer shape.
