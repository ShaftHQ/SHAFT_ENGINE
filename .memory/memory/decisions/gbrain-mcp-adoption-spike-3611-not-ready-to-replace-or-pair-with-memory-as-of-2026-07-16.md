Follow-up root-cause dig (same day): ran gbrain run_doctor, which explicitly names the fix -- an embed-backfill job -- and get_status_snapshot / get_stats, which confirm embedded_count is 0 of 734 chunks and backfill_queued/backfill_active are 0 with cycle.last_full and cycle.last_targeted both null, meaning no backfill or autopilot cycle has ever run. A broad sourceId-default embed-backfill job submission was correctly denied by the permission classifier as an unscoped change to shared infrastructure. A narrower, single-page-scoped embed job (submit_job name embed with just a slug) was accepted and queued (job id 1) but sat in waiting status with attempts_started 0 for over 20 seconds -- nothing ever picks jobs off the queue. Root cause found via the gbrain CLI (same local binary and version as the MCP server, confirmed installed under the user bun bin directory): the CLI's own help text states jobs work (the worker daemon) is Postgres only, and this brain runs on PGLite (doctor already flags queue_health as PGLite -- no queue to check) -- so no worker process can ever drain the queue in this configuration, regardless of job scope. The CLI also offers a direct synchronous alternative (embed slug, bypassing the queue entirely), but invoking it from a separate terminal failed twice, consistently, with a PGLite lock timeout error -- PGLite is single-writer/single-process, and the live MCP session already holds the database open, so no second process (including the CLI) can touch it concurrently. Net conclusion: in this specific PGLite-mode deployment, embeddings cannot be generated at all while an MCP client session is connected, via any tool reachable from inside that session -- fixing this for real requires either migrating the brain to Postgres (so jobs work can run), or invoking gbrain embed or gbrain jobs work from a separate process while no MCP session holds the lock. Neither is something to do unilaterally from inside a live MCP session; this is an infrastructure decision for whoever owns the gbrain instance, not a spike-fixable bug.
