Issue #3603 (and its follow-up #3605) identified that several informational JLabels in shaft-intellij (ShaftMcpSetupPanel, ShaftAssistantPanel, ShaftSettingsConfigurable, GuidedWorkflowPanel) call setAccessibleName(...) with a short, static, generic string even though the label's real, sometimes dynamically-updated setText(...) content carries the actual information a screen-reader user needs -- because JLabel's accessible name falls back to getText() live only when no explicit override exists, the static override permanently hides the live content from assistive tech. The adopted fix pattern (implemented in the PR for #3603, branch codex/3601-a5-and-a11y-followups): at every setText(...) call site for such a label (both the constructor/initial value AND every later live-update site -- grep the field name to find all of them, not just the constructor), also call label.getAccessibleContext().setAccessibleDescription(<the same text just set>). Leave the existing setAccessibleName(...) value completely untouched -- several of these accessible names double as findByAccessibleName(...) test selectors across ShaftPanelSetupTest/GuidedWorkflowPanelTest/ShaftSettingsConfigurableTest, so renaming them would break those tests. This is additive only, no visual/screenshot change. Fixed in #3603: GuidedWorkflowPanel#advancedHint, GuidedWorkflowPanel#recorderStatus (found beyond the issue's list), ShaftAssistantPanel's freshProjectNotice hint label, ShaftAssistantPanel#currentAgentConfiguration, ShaftAssistantPanel#captureReviewStatus (found beyond the issue's list), ShaftSettingsConfigurable#currentAgentConfiguration, ShaftMcpSetupPanel#setupSummary/#toast/#status/#recoveryStatus/#readyChecklist. Remaining same-shape instances (ShaftMcpSetupPanel's setupStatusLabel-built labels: geminiKeyStatus/upgradeDetail/mcpVersionDetail/runtimeStatus/assistStatus/recommendedAgent; ShaftSettingsConfigurable's testStatus/testRecovery and keyStatusLabel-built provider-key-status labels) were deliberately left for a bounded follow-up, tracked as GitHub issue #3605, rather than expanding this diff further.
