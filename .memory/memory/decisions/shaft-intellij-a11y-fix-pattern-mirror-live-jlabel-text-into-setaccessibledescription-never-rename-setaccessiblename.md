Issue #3603 (and its follow-up #3605) identified that several informational JLabels in shaft-intellij (ShaftMcpSetupPanel, ShaftAssistantPanel, ShaftSettingsConfigurable, GuidedWorkflowPanel) call setAccessibleName(...) with a short, static, generic string even though the label's real, sometimes dynamically-updated setText(...) content carries the actual information a screen-reader user needs -- because JLabel's accessible name falls back to getText() live only when no explicit override exists, the static override permanently hides the live content from assistive tech. The adopted fix pattern (implemented in the PR for #3603, branch codex/3601-a5-and-a11y-followups): at every setText(...) call site for such a label (both the constructor/initial value AND every later live-update site -- grep the field name to find all of them, not just the constructor), also call label.getAccessibleContext().setAccessibleDescription(<the same text just set>). Where a shared helper method is the single choke point every update runs through (e.g. showStatus(), updateStoredState()), add the mirror once inside the helper rather than duplicating it at every call site. Leave the existing setAccessibleName(...) value completely untouched -- several of these accessible names double as findByAccessibleName(...) test selectors across ShaftPanelSetupTest/GuidedWorkflowPanelTest/ShaftSettingsConfigurableTest, so renaming them would break those tests. This is additive only, no visual/screenshot change. Fixed in #3603: GuidedWorkflowPanel#advancedHint, GuidedWorkflowPanel#recorderStatus, ShaftAssistantPanel's freshProjectNotice hint label, ShaftAssistantPanel#currentAgentConfiguration, ShaftAssistantPanel#captureReviewStatus, ShaftSettingsConfigurable#currentAgentConfiguration, ShaftMcpSetupPanel#setupSummary/#toast/#status/#recoveryStatus/#readyChecklist. Fixed in #3605 (completing this pattern across the whole plugin): ShaftMcpSetupPanel#upgradeDetail/#mcpVersionDetail (single call site each) and #recommendedAgent (two call sites, constructor + updateLiveSummary()); ShaftMcpSetupPanel#runtimeStatus/#assistStatus via the shared showStatus() choke point; ShaftMcpSetupPanel#geminiKeyStatus; ShaftSettingsConfigurable#testStatus/#testRecovery (constructor reset path + 3 live-update sites); ShaftSettingsConfigurable's four provider-key-status labels (openAiKeyStatus/anthropicKeyStatus/geminiKeyStatus/githubKeyStatus) via the shared updateStoredState() choke point -- this also fixed a real pre-existing bug in keyStatusLabel(): it hard-coded a static generic description ('Shows whether a key is stored for this provider.') that never reflected the live stored/not-stored state, exactly the anti-pattern this decision addresses. No remaining same-shape instances are known across shaft-intellij as of #3605's merge.
