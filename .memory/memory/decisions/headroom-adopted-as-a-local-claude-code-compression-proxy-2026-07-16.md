Evaluated headroom (github.com/chopratejas/headroom, pip package headroom-ai, installed v0.26.0) as a local context-compression proxy after finding an unused Docker container (ghcr.io/chopratejas/headroom:0.25.0, api_requests=0, insecure 0.0.0.0:8787 bind, RW-mounted .claude/.codex/.gemini/.headroom home directories) -- that container was destroyed to reclaim resources rather than reused, since a native CLI install gives tighter control. Verified via source (not just docs) before wiring anything: headroom/proxy/helpers.py forwards authorization/x-api-key/anthropic-api-key headers through unchanged and explicitly never logs their values (pure credential pass-through, no headroom-owned API key); headroom/telemetry/beacon.py sends aggregate-only stats (tokens saved, compression ratios -- explicitly no prompts/content/PII) to a Supabase endpoint every 5 min unless disabled. Installed with: headroom install apply --preset persistent-task --runtime python --scope provider --providers manual --target claude --port 8787 --backend anthropic --mode token --no-telemetry. Deliberately omitted --memory and --code-graph (would overlap with gbrain's own knowledge/codebase-indexing role) and --learn (would auto-write AGENTS.md/CLAUDE.md without review). Chose the persistent-task preset (schtasks, user-level) over persistent-service (sc.exe, needs admin elevation) to avoid privilege escalation. Verified live: /readyz reports healthy with upstream=https://api.anthropic.com, and the actual running proxy process command line confirms --no-telemetry took effect.
