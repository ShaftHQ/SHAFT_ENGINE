CompletableFuture.supplyAsync/runAsync with no explicit Executor argument defaults to ForkJoinPool.commonPool(), a JVM-wide pool shared with the IDE and every other plugin; blocking I/O there (MCP stdio process calls, local-agent CLI runs, PasswordSafe access) can starve the whole IDE and can't be bounded, named, or shut down on plugin unload. Issue #3622 introduced com.shaft.intellij.mcp.ShaftPluginExecutor: an applicationService wrapping a fixed-size (4-8, sized off available processors) ThreadPoolExecutor with daemon shaft-mcp-worker-N threads, implementing Disposable (shutdownNow() on dispose). Issue #3623 reused it as the background executor for async PasswordSafe I/O. A regression test, NoUnboundedCommonPoolUsageTest, walks shaft-intellij/src/main for any supplyAsync/runAsync call with fewer than 2 arguments and fails the build.

Why: the shared common pool has no isolation or lifecycle tied to this plugin, so a stuck MCP/PasswordSafe call could starve unrelated IDE background work.

How to apply: every new blocking async call site in shaft-intellij must pass ShaftPluginExecutor.getInstance().executor() explicitly (or another already-bounded executor); NoUnboundedCommonPoolUsageTest will catch an omission at build time.
