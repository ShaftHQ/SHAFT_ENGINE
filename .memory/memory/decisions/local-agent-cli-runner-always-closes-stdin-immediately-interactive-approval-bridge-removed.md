PR #3373 added an interactive-approval bridge in AssistantLocalAgentRunner.run() that held stdin open past the initial prompt write for Claude Code (any client where AgentApprovalCapability.supportsInteractiveApproval()==true), waiting for a control_request/control_response round trip over Claude's stream-json OUTPUT. But the runner never adds --input-format stream-json to claudeCommand(), so in --print mode Claude reads stdin as plain text until EOF before doing anything at all -- holding stdin open therefore guaranteed every single Claude Code local-agent prompt (any content, not just tool-approval-triggering ones) hung until the 300s timeout. This was the root cause of: agent messages timing out, the skill-optimization prompt timing out, and Verbose mode showing nothing (no output was ever produced to stream). Fixed in PR #3410 by always closing stdin immediately after writing the prompt for every client (matching pre-#3373 behavior), and removing the now-dead bridge entirely (AssistantLocalAgentRunner.ApprovalRequestHandler/ApprovalDecision/enableApprovalBridge, ShaftAssistantPanel.agentApprovalHandler()/currentApprovalCapability(), and the 'Approve all SHAFT tools' toolbar checkbox that was its only UI surface). The direct-dispatch tool-approval flow (ToolApprovalService, ToolApprovalPromptPanel's inline 'Approve all tools' button, gateTool) is unrelated and was not touched -- it still works. Real interactive mid-run approval would need --input-format stream-json wired into claudeCommand() plus NDJSON-encoded prompts, verified against a live claude CLI; not attempted blind. Also fixed in the same PR: ToolApprovalPromptPanel's arguments summary was a non-wrapping JLabel and AssistantTranscriptView.widgetRow had no width cap, so the approval bubble could overflow the transcript's right edge -- now a width-capped wrapping JTextArea plus a width-capping wrapper in widgetRow. CaptureManager.stop() silently returned a stale status (looked like a successful stop) when the current process had no active recorder -- now returns an explicit warning instead. Followups needing live repro filed as issue #3409: the exact trigger for capture_code_blocks returning a bare "Done" with no code, and which of three plausible triggers orphans the browser after capture_stop.
