Issue #3662: three shaft-intellij entry points independently computed 'where does inserted SHAFT code go' and disagreed depending on which tab the user started from. shaft-intellij/src/main/java/com/shaft/intellij/java/InsertionAnchor.java now defines the single model via the pure static InsertionAnchor.choose(JavaTargetContext caretContext, String openEditorFilePath): prefer the caret's resolved Java method (insertAfter=methodName), else fall back to the open editor file (insertAfter blank), else null (no anchor). InsertionAnchorResolver.resolve(Project, Editor) is the IntelliJ-platform wrapper (PsiDocumentManager + JavaTargetContextResolver + FileEditorManager selected files) that callers actually use. Consumers: RecordShaftFlowHereAction.java:48 (caret-only, always resolvable per its update() gate -- unchanged control flow, just reuses the shape) and ShaftAssistantPanel.insertReviewIntoOpenFile() (ShaftAssistantPanel.java:3062) which previously always anchored to the open file ignoring the caret -- now prefers the caret target like RecordShaftFlowHereAction. GuidedWorkflowPanel.insertCodeAtCaret() (GuidedWorkflowPanel.java:806) deliberately does NOT go through InsertionAnchorResolver: it writes already-generated code locally via WriteCommandAction at a literal caret offset rather than asking the record-at-target MCP tool to generate+insert, so it has no 'open file, no caret' fallback case -- it stays caret-only via its existing selectedJavaEditor() gate (JavaTargetContextResolver-based). Module has no BasePlatformTestCase/PSI fixtures (see PickLocatorAtCaretActionTest's own doc comment), so InsertionAnchor.choose is covered with plain JUnit (InsertionAnchorTest) and InsertionAnchorResolver's platform wiring is intentionally left untested, matching the JavaTargetContextResolver precedent.
